# Brainstorm: Agentic Coding 검증시간 최소화

## Problem Statement

현재 병목은 **머지 후 테스트 실행**에 집중되어 있어 피드백이 늦고, 실패 발견 시 되돌리기 비용이 크다.
목표는 **테스트 단계 수를 줄이면서도** 동기화 기능과 기존 기능 회귀를 안전하게 보호하는 것이다.

## Target Users

- 1차 사용자: 개발자
- 주요 니즈: 코드 변경 직후 빠르게 안전 여부를 판단하고, 불필요한 대기 없이 병합/배포로 이어가기

## Success Criteria

- 머지 후 필수 검증 단계가 단순화된다. (예: 단일 게이트 + 야간 전체 회귀)
- PR 단계에서 실패가 조기에 검출되어 머지 후 롤백 빈도가 감소한다.
- 동기화 영역과 핵심 기존 기능 회귀는 항상 보호된다.
- 개발자가 체감하는 검증 대기 시간이 유의미하게 단축된다.

## Proposed Approach

### 1. 검증 중심을 PR 이전으로 이동 (Shift-left)

- 머지 후 실행하던 필수 테스트를 PR 필수 게이트로 이동한다.
- 머지 후에는 스모크 수준만 남기고, 전체 회귀는 주기 실행(예: 야간)으로 전환한다.

### 2. 단계 수 축소를 위한 2-Tier 게이트

- Tier 1 (필수, 빠름): lint/typecheck + 변경영향 단위/통합 테스트 + 동기화 스모크
- Tier 2 (비동기, 넓음): 전체 회귀/디바이스 조합 테스트를 스케줄링 실행
- 원칙: 배포/머지 결정은 Tier 1로, 품질 추세 관리는 Tier 2로 분리

### 3. 변경영향 기반 테스트 선택

- 파일/모듈 변경 매핑으로 관련 테스트만 우선 실행한다.
- 동기화 관련 코드 변경 시에는 동기화 보호 테스트를 강제 포함한다.
- 공통 코어 변경 시에만 확장 테스트 세트를 추가한다.

### 4. 동기화·회귀 보호를 위한 고정 안전망

- 동기화 필수 시나리오(업로드/다운로드/충돌해결/재시도) 최소 세트를 항상 실행한다.
- 기존 핵심 기능(로그 작성, 조회, 수정, 삭제) 회귀 스모크를 고정 게이트로 유지한다.
- flaky 테스트는 격리하고, 재시도 정책을 표준화해 신호 품질을 높인다.

### 5. 에이전트 출력 표준화로 리뷰/검증 압축

- 에이전트가 PR에 아래를 고정 출력한다.
- 변경 요약(영향 모듈)
- 위험 요인(동기화/회귀)
- 실행한 테스트 목록/결과
- 미실행 테스트와 이유
- 리뷰어가 재판단할 시간을 줄이고, 승인 결정을 빠르게 한다.

## Constraints

- 명시된 제약: 없음
- 운영상 권장 제약:
- 테스트 단계 감소가 커버리지 공백으로 이어지지 않도록 동기화/핵심 회귀 세트는 최소 고정
- 단계 수를 줄이더라도 실패 신호의 신뢰도(Flake 관리)는 유지

## Edge Cases

- PR 게이트를 통과했지만 환경 차이로 머지 후 실패하는 경우
- 동기화 코드 간접 영향(공통 유틸 변경)으로 실제 리스크가 누락되는 경우
- 테스트 선택 로직 오류로 필요한 테스트가 빠지는 경우
- flaky 테스트가 빠른 게이트를 반복 실패시켜 오히려 리드타임을 악화시키는 경우

## Scope

### MVP (Must-have)

- 머지 후 테스트 중 필수 단계 최소화(빠른 게이트 1개 중심)
- 변경영향 기반 테스트 선택 도입
- 동기화 필수 보호 테스트 고정
- 핵심 기존 기능 회귀 스모크 고정
- PR 템플릿에 검증 요약 표준 항목 추가

### Nice-to-have (Future)

- 테스트 소요시간/실패율 기반 자동 테스트 우선순위 조정
- 동기화 리스크 스코어링으로 동적 게이트 강화
- 에이전트 자동 triage (재현/원인 후보/권장 수정 제안)

## Open Questions

- 목표 수치 확정: “테스트 단계 감소”를 어떤 KPI로 볼지 (단계 수, 분 단위 대기시간, 리드타임)
- Tier 1 허용 최대 실행 시간 (예: 10분/15분)
- 동기화 필수 시나리오 최소 세트의 정확한 목록
- 전체 회귀 실행 주기 및 실패 시 대응 SLA
